## 3. Fixed findings (resolved in this branch)

### F-01 — Critical: Non-semantic interactive elements in dashboard lists

**WCAG criterion:** 1.3.1 Info and Relationships (A), 4.1.2 Name, Role, Value (A)  
**Non-negotiable:** Rule 7.2  
**Components:** `UpcomingAbsencesList.tsx`, `PendingValidationsList.tsx`

**Issue:** List items used `role="button"` with `tabIndex={0}` and `useNavigate` for click handling. Screen readers announced them as buttons but they navigated to a detail page — a semantic mismatch. Keyboard users could not activate them with Enter in all browsers.

**Fix:** Replaced `<li role="button">` + imperative `navigate()` with `<Link to="/absences/$absenceId">` wrapped inside a plain `<li>`. Navigation semantics are now correct and keyboard/SR behaviour follows browser defaults.

**Commit:** `fix(a11y): Replaces non-semantic <li role=button> with <Link> in dashboard lists`

---

### F-02 — Critical: Delete team action had no UI wiring (inaccessible feature)

**WCAG criterion:** 4.1.2 Name, Role, Value (A)  
**Component:** `AdminPage.tsx`

**Issue:** `TeamsTable` already implemented an `onDelete` prop with a confirmation dialog, but `AdminPage` never passed the prop. The delete functionality was unreachable from the keyboard and had no accessible confirmation flow.

**Fix:** Imported `useDeleteTeam`, added `handleDeleteTeam` handler, and passed `onDelete` to `<TeamsTable>`. An error alert is shown if the mutation fails.

**Commit:** `fix(a11y): Wires onDelete to TeamsTable in AdminPage`

---

### F-03 — High: Missing skip-to-main link in authenticated layout

**WCAG criterion:** 2.4.1 Bypass Blocks (A)  
**Component:** `AuthLayout.tsx`

**Issue:** No skip link existed. Keyboard-only users had to tab through the entire `AppNav` on every page load before reaching the page content.

**Fix:** Added a visually-hidden `<a href="#main-content" className="sr-only focus:not-sr-only …">` as the first focusable element in the layout. `<main>` was given `id="main-content"`. The brand `<span>` received `aria-hidden="true"` and the user name `<span>` received a descriptive `aria-label`.

**Commit:** `feat(a11y): Installs jest-axe and adds skip-to-main link in AuthLayout`

---

### F-04 — High: Missing ARIA live regions on async loading/error states

**WCAG criterion:** 4.1.3 Status Messages (AA)  
**Components:** `DashboardPage.tsx`, `AbsenceDetailPage.tsx`, `AuditPage.tsx`

**Issue:** Loading spinners and error messages were rendered as plain `<p>` elements with no ARIA roles. Screen readers did not announce them automatically when they appeared dynamically.

**Fix:**

- Loading paragraphs: `role="status" aria-live="polite"`
- Error paragraphs: `role="alert"`
- `<Download>` icon in `AuditPage` toolbar: `aria-hidden="true"` (decorative)

**Commit:** `fix(a11y): Adds ARIA live region roles to loading and error states`

---

### F-05 — Medium: `jest-axe` automated tests missing

**WCAG criterion:** (process quality, not a criterion)  
**Components:** `LoginPage.test.tsx`, `AdminPage.test.tsx`

**Issue:** No automated axe-based accessibility tests existed anywhere in the test suite.

**Fix:** Added `jest-axe` as a devDependency and wrote axe violation assertions for `LoginPage` (login form state) and `AdminPage` (users tab with populated data). Both tests pass with zero violations.

**Commit:** `test(a11y): Adds jest-axe automated accessibility tests to LoginPage and AdminPage`

## 4. Open findings (filed as individual issues)

### F-06 — Medium: AppNav active link uses underline as sole non-color indicator

**WCAG criterion:** 1.4.1 Use of Color (A)  
**Component:** `AppNav.tsx`  
**Issue:** [#254](https://github.com/peekabooty/let-me-out-app/issues/254)

The active navigation link relies solely on underline + colour to convey its state. An additional visual property (e.g., `font-semibold`, background pill) is required.

---

### F-07 — Medium: Dashboard shows raw English enum values for absence status

**WCAG criterion:** 3.1.1 Language of Page (A), plain-language usability  
**Component:** `UpcomingAbsencesList.tsx`  
**Issue:** [#255](https://github.com/peekabooty/let-me-out-app/issues/255)

Status strings (`accepted`, `waiting_validation`, `rejected`) are rendered verbatim. They should be mapped to translated, human-readable labels before display.

---

### F-08 — Medium: AbsenceDetailPage shows raw UUID for absence type

**WCAG criterion:** 1.3.1 Info and Relationships (A)  
**Component:** `AbsenceDetailPage.tsx`  
**Issue:** [#256](https://github.com/peekabooty/let-me-out-app/issues/256)

`absence.absenceTypeId` (a UUID) is rendered as the type label. The resolved name (e.g., "Vacaciones") must be shown instead.

---

### F-09 — Low: UnauthorizedPage missing `<main>` landmark

**WCAG criterion:** 1.3.6 Identify Purpose (AAA) / ARIA Landmarks (complementary AA)  
**Component:** `UnauthorizedPage.tsx`  
**Issue:** [#257](https://github.com/peekabooty/let-me-out-app/issues/257)

The 403 error page has no `<main>` element. Screen reader users cannot jump to the primary content directly.

---

### F-10 — Low: CalendarPage missing top-level `<h1>` heading

**WCAG criterion:** 1.3.1 Info and Relationships (A)  
**Component:** `CalendarPage.tsx`  
**Issue:** [#258](https://github.com/peekabooty/let-me-out-app/issues/258)

No `<h1>` exists at the page level. The only heading is nested inside a child component, breaking the heading hierarchy.

---

### F-11 — Medium: DashboardPage action links are `<a>` elements styled as buttons

**WCAG criterion:** 1.3.1 Info and Relationships (A), Non-negotiable 7.2  
**Component:** `DashboardPage.tsx`  
**Issue:** [#259](https://github.com/peekabooty/let-me-out-app/issues/259)

"Solicitar ausencia" and "Exportar CSV" use `<a>` elements with button styles. They should use `<Button asChild>` or a semantic `<button>` so keyboard and screen reader semantics are correct.

## 6. Recommendations for future audits

1. Expand `jest-axe` tests to all page-level components, not only `LoginPage` and `AdminPage`.
2. Add a Playwright/axe-playwright end-to-end accessibility sweep on the CI pipeline.
3. Enforce WCAG AA contrast on team colour swatches as per Non-negotiable 7.5 — validate with `@radix-ui/colors` or a design token system.
4. Establish a recurring audit cadence (quarterly) tied to the project milestone.
